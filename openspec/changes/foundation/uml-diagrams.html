Diagramas UML - foundation

Change foundation · Generado el 2026-06-24

El change foundation establece la base global ejecutable del monorepo CheckList (pnpm workspaces client + server + shared): contrato de tipos compartido, andamiaje de servidor Fastify (con GET /health, servido estatico en mismo origen, conexion y esquema SQLite, y gateway Socket.IO vacio), armazon de cliente Vite + React + Zustand, y empaquetado Docker + CI. No implementa ninguna historia de usuario ni endpoints CRUD; su objetivo es dejar el proyecto arrancable y verificable.

#Supuestos y huecos

  • Supuesto: los actores son tecnicos (Desarrollador, CI de GitHub Actions, Operador local). El change no introduce actores de negocio; el usuario final aparece en fases posteriores.
  • Supuesto: no existe maquina de estados de dominio en foundation; se propone como minima coherente el ciclo de vida del runtime del servidor (arranque), derivado de los requisitos de server-runtime.
  • Supuesto: en los diagramas de secuencia, Cliente Socket.IO y Navegador representan procesos tecnicos de verificacion del andamiaje, no usuarios reales.
  • Hueco: los contratos de eventos en shared/eventos.ts se declaran pero no se emiten en este change; aparecen como andamiaje, sin flujo de difusion (Fase 5 tiempo-real).
  • Hueco: las reglas de negocio (CRUD, progreso, reordenar, reset) estan fuera de alcance; no se diagraman aqui.

#Casos de uso

Actores tecnicos que operan sobre el andamiaje. Las relaciones include marcan pasos obligatorios del arranque del servidor; extend marca verificaciones opcionales de cierre. No hay actores de negocio en esta fase.

#Actividad

Flujo de arranque del servidor (server-runtime): carga y valida configuracion, abre SQLite y crea el esquema de forma idempotente, registra rutas, estatico y gateway, y queda escuchando. Swimlanes por componente participante.

#Secuencia 1 - Happy path: GET /health

Verificacion de salud del servidor arrancado. El endpoint responde sin depender de logica de negocio. Precondicion: la app Fastify esta escuchando.

#Secuencia 2 - Arranque del servidor con SQLite

Bootstrap completo: el entrypoint carga config, abre la base, crea el esquema idempotente y deja la app lista. Precondicion: existe .env o variables por defecto y DATABASE_URL accesible.

#Secuencia 3 - CI: build y test

Pipeline de GitHub Actions sobre push o pull request: instala pnpm, compila los tres paquetes y ejecuta los tests de andamiaje con Vitest. Precondicion: commit con el andamiaje integro.

#Estado

No hay maquina de estados de dominio en foundation. Se propone como minima coherente el ciclo de vida del runtime del servidor, derivado de los requisitos de server-runtime.
